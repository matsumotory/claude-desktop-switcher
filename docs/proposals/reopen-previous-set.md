# 前回開いていた環境をまとめて開き直す

## 背景

Claude Desktop の更新機構 Squirrel.Mac は、アプリバンドルを置き換える前に、起動中の全 Claude インスタンスの終了を待つ (ShipIt-main.m が「対象アプリの全インスタンスの終了を待つ」と明記)。CSW の各環境は同じ Claude.app を別の `--user-data-dir` で起動した別プロセスなので、同時に複数の環境を開いていると、更新を適用するにはその全部を手で終了する必要がある。2026-07-07 の実機ログでは、更新の install が 14:49 から 14:54 まで 4 回リトライされ、環境が 1 つずつ閉じられていった。さらにこの回は更新後の自動再起動が無く (`launchAfterInstallation` が false)、ユーザーは既定と研究用の 2 つを手で開き直した。ユーザーが「重複起動していると全部落とさないといけない、開き直しが面倒」と報告 (2026-07-07)。

調査の結論 (2026-07-07 の Workflow):
- 更新後の自動再起動は起きる回と起きない回があり、Claude 側の更新経路で決まる。起きる回でも起動引数は引き継がれず、既定のデータ領域で開く (v0.21.0 の乗り移り検知が観測している現象)。
- したがって「乗り移り検知に一括復元をぶら下げる」設計は、今回の動機ケース (自動再起動なし、約 4 分の段階的終了) では発火しない。困りごとの本体は「N 個を手で開き直すクリック数」であって更新検知ではない。
- そこで、更新検知にもバナーにも自動起動にも紐付けない、手動の「前回開いていた環境を開き直す」を主軸にする。

## ゴール

- 直前まで同時に開いていた環境の集合を CSW が覚え、設定ウィンドウの案内から 1 回の操作でまとめて開き直せるようにする。更新でも、Mac の再起動でも、集中のため全部閉じたあとでも、同じ導線で戻せる。
- 自動では何も起動しない。ユーザーが押したときだけ起動する。CSW から Claude を終了させない。
- いま動いている環境は開き直しの対象から必ず除外する。共有・コピー・既定の環境は一度に 1 つしか安全に開けないため、別の Claude が動いていて開けない環境は、勝手に起動せず、先に動いている Claude を終了するよう正直に案内する。

## 対象ユーザーと規模

複数の環境を同時に開いて使う全ユーザー。コアの純関数 1 つと小さな状態ファイル 1 つ、監視ループへの数十行、案内バナー 1 枚で、困りごとの頻度 (更新は数日に 1 回) に見合う軽さに収める。

## 設計

### 集合の記録 (高水位ユニオン)

トレイの 3 秒監視に、集合の記録を足す。

- `session_union`: 前回すべて閉じてから、同時に開いていた環境名の和集合。毎ティック `running_profile_names_from` の結果を高い方に更新する。段階的に閉じても全部を拾えるよう、スナップショット 1 点ではなくユニオンで持つ。
- すべて閉じたティック (running が空) で、`session_union` が空でなければ、それを復元候補として CSW 自身のフォルダに保存し、`session_union` を空に戻す。保存は最終起動の記録と同じ atomic 書き込み (temp+rename)。
- 復元候補の保存は「すべて閉じた瞬間」の 1 回だけなので、書き込みは低頻度。CSW の再起動や Mac の再起動をまたいで残る。
- 限界の明示: 更新後に自動再起動が速く、監視の 3 秒間隔が「すべて閉じた瞬間」を捉え損ねた場合、復元候補は前回すべて閉じたときの集合になる。案内には実際の環境名を出すので、ユーザーが見て判断できる。

### 開き直しの安全性 (core の純関数)

`plan_relaunch(recorded, running, envs) -> RelaunchPlan` を core に置き、RED テストで固める。

- いま動いている環境は、既定も完全分離も例外なく最初に全部除外する (`open -n` は毎回新プロセスを起こすため、動いているものを再起動すると同じデータで 2 重起動になる)。
- 完全分離の環境は何個でも並べて起動できる (`additional`)。
- 既定・共有・コピーの環境は一度に 1 つしか安全に開けない。何も動いていなければ 1 つを active として起動する。別の Claude が動いていれば起動できないので、`blocked` として別枠で返し、UI は先に動いている Claude を終了するよう案内する。複数ある場合も 1 つだけ active にし、残りは blocked。
- 起動の順序は、active を先に (`switch_to` は Claude 起動中を拒否するため、何も動いていない状態で先に)、その後で完全分離を追加起動する。

### UI

- 復元候補に「いま動いていない環境」が 1 つ以上あるときだけ、設定ウィンドウの上部にバナーを出す。実際の環境名を並べ、「開き直す」と「閉じる」を置く。
- 「開き直す」は復元を実行し、起動したものと、別の Claude の終了が要るものをトーストで知らせ、バナーを消す。
- 「閉じる」は復元候補を消す (再度の催促をしない)。
- v0.21.0 の乗り移りバナーが出ているときは、こちらのバナーを出さない。案内を 2 枚重ねない。乗り移りが解消したあと、復元候補が残っていればこちらを出す。

## リスクと対応

- 意図しない開き直し: 自動では何も起動しない。ユーザーがバナーの「開き直す」を押したときだけ起動する。バナーは実際の環境名を出すので盲目的な起動にならない。
- 更新の適用中に起動して壊す: Squirrel はバンドル差し替え中にそのバンドルから起動されると更新を中止・破損し得る。CSW は自動起動せず、ユーザーが押したときだけ起動するので、起動のタイミングは今の手動再起動と同じで、新しい危険を増やさない。
- 2 重起動: 復元は必ず現在の起動集合との差分だけを起動する。既定も完全分離も除外する。
- 既定常駐で集合が確定しない: 既存の Claude を常に開いている運用では running が空に戻らず、復元候補が更新されないことがある。第 1 段階では「すべて閉じたら確定」の単純な規則にし、限界を docs に書く。
- 揮発でなく永続: 乗り移りの記録と違い、復元は「前回のセットを開き直す」明示の操作で、古くても環境名が見えるので誤解を生まない。再起動をまたいで役立つよう永続にする。存在しなくなった環境は `list_profiles` で寛容に除外する。

## スコープ外 (混ぜない)

- 更新の兆候 (ShipIt のログや Claude.app のバージョン) の読み取り。制約 3 のグレーゾーンに踏み込まないため、第 1 段階では扱わない
- 乗り移りバナーへの畳み込みと、更新後の自動提示 (第 2 段階。ユーザーが望んだときだけ)
- CSW からの Claude の終了 (ユーザー判断で恒久見送り)
