# トリプルAI開発体制と言語ルール

本プロジェクトは、認知科学の **System 1 / System 2** モデルに着想を得た3つのAIエージェントの分業体制で開発を進めます。

## トリプルAI開発体制

| エージェント | 思考モデル | 役割 |
|---|---|---|
| **Antigravity** | System 1（直感・高速） | ローカル環境で新機能の高速実装。`// TODO: Jules:` で技術的負債をマークし先に進む |
| **Jules** | System 2（分析・熟慮） | クラウド環境で非同期にコード品質を改善。TODO回収・テスト追加・リファクタリングをPRで提案 |
| **Gemini Code Assist** | System 2（分析・熟慮） | GitHub上でPRの自動コードレビュー。`/gemini review` コマンドで起動 |

- 各エージェントの詳細な責務は `core_agent_roles` スキルを参照
- セッションの開始/終了フローとタスク委譲は `core_ai_delegation` スキルを参照
- PRレビューのプロトコルは `core_pr_review_cycle` スキルを参照

### 計画と Blueprint・README 管理
- 大幅な変更や新機能追加の際は、コーディング前に必ず `PLANNING` モードに入る
- **機能を変えたら、その仕様の正典である `docs/` 直下の該当テーマ別ファイルを必ず更新し、プロダクト概要が動くなら `README.md`、全体像の位置づけが動くなら `blueprint.md`（全体像の地図）も合わせて更新する。** 詳細仕様は `blueprint.md` に二重化せず、テーマ別ファイルを正典とする。

### ブランチ戦略
- **すべての変更（コード・ドキュメント・plan ファイル含む）は必ず feature ブランチで作業し、CI パス後に PR 経由でマージすること。`main` への直接 push は GitHub branch protection rule で物理的に阻止されており例外はない。詳細は [AGENTS.md](../../../AGENTS.md) §4.4。**
- **デモデプロイ**: `demo` ブランチに `main` をマージして push → Firebase App Hosting が自動デプロイ。手順は [DEMO_DEPLOY.md](../../../DEMO_DEPLOY.md) 参照

## 言語ルール

### 日本語コミュニケーション（最重要）
**AIエージェントは、すべてのコミュニケーション・ドキュメントを日本語で記述しなければならない。**
- ユーザーとの会話、計画書、PR、Issue、コミットメッセージ
- Technical terms やコードスニペットは英語で構わない

### UIテキストは日本語直書き
- 国内向けサービスのため、i18n対応は行わない。**新規UIテキストはTSX/TSに日本語で直書きする**
- `/dictionaries/` の既存辞書参照は動作するのでそのまま残し、漸進的に直書きへ移行する
- 英語辞書（`en/`）の同期更新は不要
