# 開発体制と言語ルール (Claude Code)

本プロジェクトは Claude Code の単一エージェントセッションで開発する。重い調査・レビュー・並列作業は、メインエージェントが必要に応じてサブエージェントを起動して分担する。

## エージェント体制

| 役割 | 実体 | 使いどころ |
|---|---|---|
| **メインエージェント** | 現在のセッション | 仕様合意・実装・コミット・ユーザーとの対話。最終責任を持つ |
| **調査/レビュー用サブエージェント** | `Task`/`Agent` ツール | 広い検索、複数ファイル横断の読解、独立した観点でのレビューを並列で委譲。結論だけ受け取る |
| **多段オーケストレーション** | `Workflow` ツール | 大規模な監査・移行・網羅レビューなど、決定的なファンアウト/検証が要るとき |

- 独立した調査は 1 メッセージで複数のサブエージェントを同時起動して並列化する。
- サブエージェントの報告 (特に「PASS」「問題なし」) を鵜呑みにしない。重要な結論は一次情報 (コード/コマンド出力) を自分で確認する (`spec_first_development` 参照)。
- 技術的負債を見つけたら素の `// TODO:` でマークし、必要なら別タスクとして切り出す。

### 計画とドキュメント同期
- 大幅な変更や新機能の前に、まず方針を整理する (必要なら plan モード)。
- **機能を変えたら、その仕様の正典である `docs/SPECIFICATION.md` を必ず更新し、ユーザー向け挙動が動くなら `docs/USER_GUIDE.md` / `docs/USER_GUIDE_EN.md`、プロダクト概要が動くなら `README.md` も合わせて更新する。** 実装とドキュメントの乖離を残さない。
- LP (`website/index.html` と `website/ja/index.html`) は構造変更を両言語同時に反映する (CLAUDE.md の Multi-language Sync)。

### ブランチ戦略
- **すべての変更 (コード・ドキュメント含む) は必ずトピックブランチ (`feat/*` `fix/*` `docs/*` `refactor/*`) で作業し、CI (Test ワークフロー) がパスしてから PR 経由でマージする。`main` への直接 push は禁止。** 詳細は [CLAUDE.md](../../../CLAUDE.md) の Branch Strategy Enforcement。

## 言語ルール

### 日本語コミュニケーション (最重要)
**ユーザーとの会話・計画・PR・Issue・コミットメッセージはすべて日本語で記述する。**
- technical term やコードスニペット、Conventional Commits の `type(scope)` プレフィックスは英語で構わない。

### ユーザー向けテキスト
- ユーザー向け文言 (CLI 出力・Tauri UI・ドキュメント・LP) は日本語を基本とする。英語版ドキュメント/LP (`*_EN.md` / `website/index.html`) は別途英語で保守する。
- 装飾目的の絵文字 (チェックマーク・旗・アイコン類) はユーザーが明示的に要求しない限り使わない (CLAUDE.md の Tone & Copywriting)。

## サブエージェント運用の原則

サブエージェント (`Task`/`Agent`) と `Workflow` を安全に使うための実務ルールをまとめます。上の「エージェント体制」で示した「結論を鵜呑みにしない」を、具体的な起動条件・検証手順まで落とし込んだものです。

### 現在日付と最新一次情報を必ず渡す
調査・研究・仕様確認を任せるエージェントには、プロンプトの先頭で現在日付を明示し、訓練データの記憶で即答させず、最新の一次情報 (`WebSearch`/`WebFetch`、公式 docs・リリースノート) で確認してから答えるよう指示します。

- 日付やバージョンに依存する主張は、必ず「〜時点」と時点を添えて報告させます。時点のない断定は陳腐化を検知できません。
- 複数のエージェントを同時に起動するときは、全員に同じ日付と「最新を取りに行く」指示を渡します。1 つだけに書いても、指示を渡し忘れたエージェントが古い記憶で答えて全体の結論を汚します。
- 受け取った結論のうち、日付・バージョン・外部仕様に関わる部分は、根拠 URL とアクセス時点が示されているかを点検してから採用します。

### システム固有の事実は実機検証を優先する
CLI フラグの対応可否・ファイルパスの存在・特定バージョンの挙動といった、この環境に固有の事実は、エージェントの Web 調査よりも実機検証 (`man`・実ファイルの `Read`・実コマンドの実行) を優先します。

- Web 調査は一般論としては正しくても、手元の環境・バージョンでは異なることがあります。エージェントの結論は、自分の手で取った ground truth と突き合わせてから採用します。
- 「対応している/存在する」という肯定的な結論ほど、実際に叩いて確かめます。存在しないフラグやパスを前提に実装を進めると、後段でまとめて破綻します。

### 読み取り専用調査と編集を伴う検証を分ける
既定の full-tool エージェントは `Edit`/`Write` を持ち、検証のためにソースを書き換えることがあります。同じ作業ツリーで起動すると、その書き換えが呼び出し元の作業ツリーを汚します。

- 読むだけの調査は、`Explore` の agentType を使うか、プロンプトで「ファイルを編集しない・検証は説明のみで行う」と明示的に縛ります。
- 依存の bump を実ビルドで確かめるなど、編集を伴う検証が必要なときは、隔離した worktree で実行し、現在の作業ツリーと切り離します。
- workflow やサブエージェントを動かしている間、同じ worktree で自分の `git add`/`commit` やビルドを並行して走らせません。ビルド (例: `cargo tauri build`) 自体が manifest を正規化して差分を生むため、どちらの変更か切り分けられなくなります。派生タスクの隔離手順は `core_worktree_for_derived_tasks` を参照します。

### critic の repo 事実系 finding は現行 worktree で再検証する
`Workflow`/サブエージェントは、起動時の cwd (= その worktree) を grep して事実を判定します。起動 worktree が現在の作業 worktree より古いと、最新で追加された機能・スキル・docs が「存在しない」「repo は逆のことを主張している」という偽の blocker として返ってきます。

- critic が返す repo 事実系の指摘 (「存在しない」「docs は逆」「裏が取れない」) は、鵜呑みにせず、現在の作業 worktree で自分で `Grep` や `gh` を使って再検証してから採否を決めます。
- 文章・論理・可読性に関する指摘は cwd に依存しないため、そのまま採用できます。事実系と論理系を切り分けて扱います。

### 実装・修正・検証は Workflow のフィードバックループで組む
重い実装は、`Workflow` の中で implement → build/test/clippy → 診断 → fix → 再検証 を繰り返すループとして組み、緑に収束させてから受け取ります。

- メインエージェントの役割は、方針の決定・コミット/PR・ユーザーとの対話・結果のレビューに絞ります。実装そのものは workflow に委譲し、メインループでは各サブ判断ごとに手を止めません。
- ループの終了条件は「ビルド・テスト・lint がすべて緑」に置き、緑になった根拠 (コマンド出力) を確認してから完了とします。検証の順序とエビデンスの扱いは `spec_first_development` に従います。

#### ループの安全装置
収束しないまま修正を重ね続けるのは自走ではありません。反復には次の上限と隔離を課します。
- **修正回数の上限**: 同一の失敗に対する自動修正は 3 回までとし、超えたらループを止めて報告します。
- **試行の隔離**: 試行錯誤は worktree か作業ブランチに閉じ、成功した最終状態だけを本流へ反映します。
- **悪化時の巻き戻し**: 2 回目の修正が 1 回目より状態を悪化させたら、修正を重ねず直前の良好な状態へ revert します。
- **未収束の報告**: 上限に達しても収束しないときは、成功を装わず、試した内容と残っている問題を明示して報告します。
